# Weekly Summary

用于生成偏简洁的实习生周报。重点不是写得华丽，而是把本周做了什么、推进到哪里、遇到了什么问题、下周准备做什么说清楚。

## 1. 默认目标

- 输出简洁、自然、像真实实习生会提交的周报
- 以事实为主，不夸大影响，不把讨论写成结果
- 优先按“项目 -> 每次 push / 推送批次 -> 周总结”组织内容
- 将 Codex 对话记录作为辅助材料，用来补充背景、问题定位、方案取舍，不替代代码和提交事实
- 周报正文只保留对领导有价值的工作结果、推进情况和下周计划；AI 在执行过程中遇到的取证缺口、收集失误、工具限制等，只需要单独告诉用户，不写进提交版周报
- 产出文件默认只写入用户指定的目标路径，不在 skill 目录下额外生成 `week1.md`、`draft.md` 之类的中间产物，除非用户明确要求保留草稿

## 2. 先确认的信息

如果用户没有给全，优先补齐这些信息：

1. 周报时间范围
2. 需要汇总的项目路径或仓库列表
3. 用户在 Git 中的身份标识
   - 用户名
   - 邮箱
   - 可能使用过的 author name
4. Codex 对话记录来源
   - 项目内导出的 markdown/json
   - 当前线程
   - 用户额外提供的聊天记录

若信息不全但可以继续，先基于已有材料产出草稿，并明确说明哪些部分是待补充项。

## 3. 数据收集顺序

### 3.1 Git 提交是主证据

对每个项目，优先收集当前周内与用户身份匹配的提交：

- `git log --author=... --since=... --until=...`
- 必要时结合 `git show --stat` 或小范围 diff 理解提交内容
- 不只抄 commit message，要提炼“动作 -> 结果/进展”

### 3.2 Push 信息优先真实，其次近似

用户希望按“每次 push”汇总时，按以下优先级处理：

1. 如果项目平台或导出记录里有明确 push 事件，按真实 push 汇总
2. 如果只有本地 git 提交，没有可靠 push 事件，不要编造“真实 push”
3. 此时降级为“推送批次”或“提交批次”汇总，并明确标注为近似分组

可接受的近似分组方式：

- 同一项目
- 同一分支
- 相近时间的一组连续提交
- 明显属于同一小任务的一组提交

分组后仍然要保留“这次主要做了什么”的可读总结，不要变成逐条 commit 清单。

### 3.3 Codex 对话记录是辅助证据

读取项目相关的 Codex 对话记录时，只提取这些内容：

- 本周在做什么任务
- 为什么这样改
- 遇到的问题和排查方向
- 已确认的限制、风险、待办

不要把下面这些内容直接写成“已完成”：

- 方案讨论
- 想法草稿
- 未落地的建议
- 只有对话、没有代码或提交支撑的结果

## 4. 项目级整理方法

每个项目按下面顺序整理：

1. 项目一句话背景
2. 本周与用户相关的 push / 推送批次列表
3. 每个 push / 批次的摘要
4. 结合 Codex 对话补充问题背景或推进过程
5. 项目当前状态

每个 push / 推送批次至少回答这三个问题：

1. 做了什么
2. 产出了什么
3. 目前到哪一步

如果信息足够，优先写成这种粒度：

`05-14 Push 2：补齐登录页表单校验和错误提示，联调后可以覆盖空值与非法格式场景，页面交互已基本跑通。`

而不是：

`修改了登录页。`

## 5. 周报写法要求

写作质量参考高质量总结类 skill，但这里默认采用更轻量的实习生风格。

### 5.1 内容要求

- 先写结果，再写动作
- 能量化就量化，但没有数据时不要硬编
- 一个工作项最多 1 到 3 句话
- 工作少的时候允许简写，不强行写复杂
- 问题描述尽量写清“现象 -> 原因/猜测 -> 当前处理状态”

### 5.2 风格要求

- 语气自然、朴素、可靠
- 避免空话，如“持续推进”“稳步开展”
- 避免夸张表述，如“显著提升”“全面优化”，除非证据明确
- 对实习生任务可以用“协助完成”“参与整理”“跟进排查”“完成基础开发”

### 5.3 真实性要求

- 不把对话推测写成事实
- 不把未合并、未验证的修改写成上线结果
- 不把他人提交算到用户头上
- 看不出成果归属时，要明确写“参与支持”而不是“独立完成”
- 不把 AI 执行过程中的问题写进周报，例如：取证口径失误、临时没查到记录、工具权限限制、技能本身的工作说明等

## 6. 输出结构

默认输出使用简洁版周报，项目较多、信息较杂时可换成复杂版。模板见 [references/templates.md](references/templates.md)。

默认推荐结构：

1. 本周工作概览
2. 按项目进展
3. 个人思考
4. 下周计划

其中“按项目进展”部分需要体现 push / 推送批次维度。

### 6.1 关于个人思考

“个人思考”默认由用户自己写，AI 不应直接编造个人感悟、态度变化或成长体验。

AI 在这一部分的职责是提供辅助，而不是代写经历本身。可以提供：

- 本周值得复盘的 1 到 3 个问题
- 从任务中可以提炼出的能力方向
- 下周可以继续加强的技术点或协作点
- 几种更自然、不过度包装的表达方式

如果用户明确要求补全草稿，可以基于本周事实给出“可修改初稿”，但要满足：

- 语气克制，像个人记录，不像总结表彰
- 只围绕本周真实工作展开
- 避免写空泛成长，如“收获很多”“能力提升明显”
- 避免把 AI 推测写成用户已经确认的想法

## 7. 推荐输出策略

### 7.1 有多个项目时

- 先给一个 2 到 4 行的总览
- 再分项目写
- 每个项目下按 push / 推送批次列出

### 7.2 工作量较少时

- 不强行凑满
- 可以把同一项目的一周小改动合并成 1 到 2 个批次总结
- 重点写清楚参与内容和当前进度

### 7.3 信息不足时

直接向用户说明缺口，但不要把这类说明写进最终提交版周报。例如：

- `未拿到可靠 push 记录，以下按提交批次近似汇总`
- `Codex 对话仅作为背景补充，完成度以 git 提交为准`
- `个人思考部分建议由用户自行补充，以下仅提供可展开的方向`

## 8. 执行时的建议命令

下面这些命令适合用来收集材料，按需使用即可：

```bash
git log --since="2026-05-12" --until="2026-05-18 23:59:59" --author="your-name|your-email" --date=short --pretty=format:"%h%x09%ad%x09%s"
```

```bash
git show --stat <commit>
```

```bash
rg -n "Codex|需求|修复|计划|问题|TODO" <project-path>
```

如果要从对话导出或 markdown 记录里抽辅助信息，优先提炼任务、问题、结论，不要大段照抄原文。

## 9. 最终产出标准

最终周报应满足：

- 能看出本周主要做了哪些事
- 能区分不同项目
- 能看到每次 push 或推送批次对应的阶段成果
- 能看出当前推进状态和下周计划
- 个人思考部分真实、简短，并且与本周实际工作相关
- 整体简洁，不堆术语，不像绩效自夸材料
- 不包含 AI 执行过程中的说明性话术、取证过程问题或面向用户的解释性备注
